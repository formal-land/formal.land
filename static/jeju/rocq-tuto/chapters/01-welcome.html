What is Formal Verification?

Imagine you're building a bridge. You wouldn't just hope it doesn't collapse — you'd use mathematics and physics to prove it will hold. Formal verification does the same thing for software!

Rocq (formerly known as Coq) is a proof assistant — a tool that helps you write mathematical proofs that a computer can verify. When Rocq says your proof is correct, you can be absolutely certain it's valid.

Don't worry if this seems abstract! We'll start with simple examples, and soon you'll be writing proofs like a pro!

Why Do Proofs Matter?

Bugs in critical systems can have serious consequences:

  • Ariane 5 rocket (1996): A software bug caused a $370 million rocket to explode
  • Therac-25 (1985-87): Software errors in radiation therapy machines caused patient deaths
  • Knight Capital (2012): A trading software bug lost $440 million in 45 minutes

Formal verification can prevent these disasters by mathematically proving that software behaves correctly.

Your First Look at Rocq

Let's see what Rocq code looks like:

first_proof.v 
(* This is a comment in Rocq *)

(* We declare a theorem and its type *)
Theorem my_first_theorem : True.
Proof.
  (* True is always provable using 'I' *)
  exact I.
Qed.

(* A slightly more interesting proof *)
Theorem identity : forall A : Prop, A -> A.
Proof.
  intro A.    (* Introduce A into our context *)
  intro H.    (* Introduce hypothesis H : A *)
  exact H.    (* Our goal is A, and we have H : A *)
Qed.

Let's break this down:

  • Theorem — Declares something we want to prove
  • Proof — Starts the proof mode
  • Qed — Finishes the proof (Latin: "quod erat demonstrandum")
  • intro, exact — These are tactics that transform our proof state

What is I?

I is the proof (or "constructor") of True. In Rocq, every proposition that can be proved has at least one proof term. True is the simplest provable proposition, and I is its unique proof. When your goal is True, you use exact I to complete the proof!

Rocq Comments

Comments in Rocq are written between (* ... *). They can span multiple lines and even be nested! This is different from languages like C (/* */) or Python (#).

Type Annotations

In Rocq, we often see expressions like A : Prop or n : nat. The colon : means "has type":

  • A : Prop — "A has type Prop" (A is a proposition)
  • n : nat — "n has type nat" (n is a natural number)
  • H : A — "H has type A" (H is evidence/proof of A)

Type annotations appear in many places: after variable names in forall bindings, in function parameters, and in hypotheses during proofs.

Understanding Types: forall and ->

Let's decode the type forall A : Prop, A -> A:

What is Prop?

Prop is the universe of propositions — things that can be true or false. Examples include "2 + 2 = 4" or "all even numbers greater than 2 are the sum of two primes".

What does forall A : Prop mean?

forall means "for all" or "for any". The expression forall A : Prop means "for any proposition A". It's universal quantification — the statement must hold no matter what proposition you choose for A.

Think of it like a function that takes a proposition as input. When we write intro A, we're saying "let A be some arbitrary proposition" — we don't care which specific proposition, the proof will work for all of them.

What does -> mean?

The arrow -> represents implication: "if ... then ...".

  • A -> B means "if A, then B" (or "A implies B")
  • A -> A means "if A is true, then A is true" — obviously correct!
  • A -> B -> C means "if A, then if B, then C" (arrows associate to the right)

In Rocq, implication and function types are the same thing! This is part of the Curry-Howard correspondence, which we'll explore more in Chapter 2.

Reading Type Signatures

forall A : Prop, A -> A reads as: "For any proposition A, if A is true, then A is true." This is the identity — given evidence of A, we return that same evidence!

Understanding the Proof State

When you're in proof mode, Rocq shows you the current proof state:

🔮 Proof State
A : Prop
H : A
Goal (1 of 1)
A
  • Hypotheses (above the line): Things we know or have assumed
  • Goal (below the line): What we need to prove

Your job as a proof writer is to use tactics that transform the proof state until the goal becomes something trivially true (or matches a hypothesis exactly).

Tactic Reference

Here are the tactics we've seen so far:

intro H

When your goal is A -> B, this tactic moves A into your hypotheses with name H, and your new goal becomes B. It also works for forall x : T, ..., introducing x as a variable.

exact H

When your goal exactly matches a hypothesis H, this solves the goal. The proof is complete if H has exactly the type of the goal.

Common mistake: If exact H fails, check that the types match exactly. For example, if your goal is B and you have H : A -> B, you need apply H instead — H is an implication, not a direct proof of B.

apply H

When you have H : A -> B and your goal is B, apply H changes your goal to A. You're reasoning backwards: "to prove B, it suffices to prove A".

Proof.
  intro A.
  intro B.
  intro HA.      (* HA : A *)
  intro HAB.     (* HAB : A -> B *)
