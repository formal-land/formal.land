The Check Command

Check asks Rocq to tell you the type of an expression without evaluating it. It's a useful exploration tool:

check_command.v 
Check I.             (* I : True          - I is a proof of True *)
Check True.          (* True : Prop       - True is a proposition *)
Check nat.           (* nat : Set         - nat is a data type *)
Check 42.            (* 42 : nat          - 42 is a natural number *)
Check (2 + 3).       (* 2 + 3 : nat       - an expression *)

(* Check doesn't compute - it just shows the type *)
(* Use Compute to actually evaluate expressions *)

Check is invaluable when you want to understand what type a term has, especially when working with unfamiliar functions or complex expressions.

Lambda Expressions: fun

The fun keyword creates anonymous functions (also called lambdas). The syntax is fun x => body, where x is the parameter and body is an expression using x:

fun_syntax.v 
(* A function that takes x and returns x *)
Check (fun x => x).              (* ?A -> ?A *)

(* A function that takes a nat and adds 1 *)
Check (fun x => x + 1).          (* nat -> nat *)

(* Multiple parameters *)
Check (fun x y => x + y).        (* nat -> nat -> nat *)

(* With explicit type annotation *)
Check (fun (x : nat) => x * 2).  (* nat -> nat *)

(* A function from True to True *)
Check (fun (H : True) => H).     (* True -> True *)

In the Curry-Howard correspondence, a function fun (H : A) => ... is the same as a proof that starts with "assume A". This is why fun (H : True) => H has type True -> True — it's both a function AND a proof of the implication!

Propositions as Types

In Rocq, there's a beautiful idea: propositions are types, and proofs are programs. This is called the Curry-Howard correspondence.

  • A proposition P is a type
  • A proof of P is a term of type P
  • An implication A -> B is a function type
curry_howard.v 
(* True has exactly one proof: I *)
Check I.       (* I : True *)

(* An implication A -> B is like a function type *)
Check (fun (H : True) => H).   (* True -> True *)

(* A proof of A -> B is a function that transforms
   evidence of A into evidence of B *)

The Core Tactics

intro vs intros

When your goal is A -> B, intro H moves A into your hypotheses as H.

What's the difference?

  • intro H — Introduces exactly one thing with name H
  • intros H1 H2 H3 — Introduces multiple things with the given names
  • intros (no names) — Introduces everything possible with auto-generated names

Using explicit names like intros A B HA HB makes proofs more readable than relying on auto-generated names.

Before intro H
A -> B

⬇️ intro H.

After intro H
H : A
B

exact

When your goal exactly matches a hypothesis, use exact H.

apply

If you have H : A -> B and your goal is B, then apply H changes the goal to A.

assumption

Like exact, but Rocq finds the matching hypothesis automatically.

Examples

examples.v 
(* Identity: proving A -> A *)
Theorem identity : forall A : Prop, A -> A.
Proof.
  intro A.      (* Introduce the type A *)
  intro H.      (* Introduce hypothesis H : A *)
  exact H.      (* Goal is A, we have H : A *)
Qed.

(* Composition of implications *)
Theorem impl_trans : forall A B C : Prop,
  (A -> B) -> (B -> C) -> (A -> C).
Proof.
  intros A B C HAB HBC HA.
  apply HBC.    (* Now we need B *)
  apply HAB.    (* Now we need A *)
  exact HA.
Qed.

(* The K combinator *)
Theorem K : forall A B : Prop, A -> B -> A.
Proof.
  intros A B HA HB.
  exact HA.
Qed.

(* The S combinator *)
Theorem S : forall A B C : Prop,
  (A -> B -> C) -> (A -> B) -> A -> C.
Proof.
  intros A B C HABC HAB HA.
  apply HABC.
  - exact HA.
  - apply HAB. exact HA.
Qed.

Bullet Points for Subgoals

When a tactic creates multiple subgoals (like apply on a hypothesis with multiple arguments), use bullet points (-, +, or *) to clearly mark each subgoal's proof. This keeps your proofs organized and readable!

More Tactics

auto

Automatically searches for a proof using introduction rules and hypotheses.

assert

Prove a helper lemma in the middle of your proof.

auto_assert.v 
(* auto can solve many simple goals *)
Theorem auto_example : forall A B C : Prop,
  A -> (A -> B) -> (B -> C) -> C.
Proof.
  auto.
Qed.

(* Using assert for intermediate lemmas *)
Theorem using_assert : forall A B C : Prop,
  (A -> B) -> (B -> C) -> A -> C.
Proof.
  intros A B C HAB HBC HA.
  assert (HB : B).
  { apply HAB. exact HA. }
  apply HBC. exact HB.
Qed.

Curly Braces { } for Goal Focusing

The curly braces { } focus on a single subgoal. Everything inside the braces must fully solve that one subgoal before you continue. This keeps your proofs organized and makes errors easier to find!

Tactic Combinators

The Semicolon ;

The semicolon chains tactics together. tactic1; tactic2 runs tactic1, then runs tactic2 on all resulting subgoals.

semicolon.v 
(* Without semicolon - more verbose *)
Theorem and_comm_verbose : forall A B : Prop, A /\ B -> B /\ A.
Proof.
  intros A B [HA HB].
  split.
  - exact HB.
  - exact HA.
Qed.

(* With semicolon - more concise *)
Theorem and_comm_concise : forall A B : Prop, A /\ B -> B /\ A.
Proof.
  intros A B [HA HB].
  split; assumption.    (* assumption is tried on BOTH subgoals *)
Qed.

The semicolon is especially useful when the same tactic works on multiple subgoals. In the example above, split creates two subgoals (prove B and prove A), and assumption solves both since we have HA : A and HB : B.

What auto Can and Can't Do

auto is powerful but limited. It can:

  • Chain together intro and apply
  • Use hypotheses from the context
  • Try simple combinations up to a limited depth

It cannot: do induction, rewrite with equations, destruct data types, or use tactics like simpl. For anything beyond basic propositional logic, you'll need to guide the proof yourself.

Exercise 2.1: Chain of Implications

exercise2_1.v
Theorem chain : forall A B C D : Prop,
  (A -> B) -> (B -> C) -> (C -> D) -> A -> D.
Proof.
  (* Your proof here! *)

Qed.

Strategy: Use intros to introduce everything, then apply the implications in reverse order. To prove D, you need C (apply the C→D implication). To prove C, you need B. To prove B, you need A — and you have A!

Alternatively, try auto after intros — it can chain simple implications automatically.

Exercise 2.2: Flip Arguments

exercise2_2.v
Theorem flip : forall A B C : Prop,
  (A -> B -> C) -> (B -> A -> C).
Proof.
  (* Your proof here! *)

Qed.

Strategy: After intros A B C HABC HB HA, you have:

  • HABC : A -> B -> C
  • HB : B
  • HA : A

Your goal is C. Apply HABC — it needs A then B, and you have both (in the opposite order you received them)!

Summary

  • Propositions are types, proofs are terms
  • intro / intros — introduce hypotheses
  • exact — use an exact matching hypothesis
  • apply — use an implication backward
  • assumption — let Rocq find the hypothesis
  • auto — automatic proof search
  • assert — prove intermediate lemmas
