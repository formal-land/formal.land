Inductive Types

In Rocq, we define types by specifying how to construct their values. Natural numbers are defined:

nat.v
(* How nat is defined in Rocq *)
Inductive nat : Type :=
  | O : nat         (* Zero is a natural number *)
  | S : nat -> nat. (* Successor: if n is nat, S n is nat *)

(* Examples:
   0 = O
   1 = S O
   2 = S (S O)
   3 = S (S (S O))
*)

Jeju says:

Think of S as "+1". So S (S (S O)) is "0 + 1 + 1 + 1 = 3"!

Type vs Prop

You may notice nat : Type here, while earlier we saw A : Prop. What's the difference?

  • Prop — The universe of propositions (logical statements that are true or false)
  • Type — The universe of data types (computational things like numbers, lists, etc.)

nat is a data type — we compute with natural numbers. True, False, and A /\ B are propositions — we prove them. The Curry-Howard correspondence connects these worlds: proving a proposition is like constructing a term of that type!

Pattern Matching

We examine natural numbers using match:

pattern_matching.v
(* Check if a number is zero *)
Definition is_zero (n : nat) : bool :=
  match n with
  | O => true
  | S _ => false
  end.

(* Predecessor function *)
Definition pred (n : nat) : nat :=
  match n with
  | O => O           (* pred 0 = 0 *)
  | S m => m         (* pred (S m) = m *)
  end.

Compute pred 5.      (* = 4 *)
Compute pred 0.      (* = 0 *)

The Wildcard _

The underscore _ is a wildcard pattern that matches anything but doesn't bind a name. Use it when you need to match a case but don't care about the value:

  • S _ — matches any successor, we don't need the predecessor's value
  • S m — matches any successor AND names the predecessor m

Using _ is good practice: it signals "this value is intentionally unused" and avoids compiler warnings about unused variables.

The Compute Command

Compute evaluates an expression and shows the result. Unlike Check (which shows types), Compute actually calculates:

compute_command.v
Compute 2 + 3.         (* = 5 : nat *)
Compute pred 5.        (* = 4 : nat *)
Compute 10 - 7.        (* = 3 : nat *)

(* Compare with Check *)
Check 2 + 3.           (* 2 + 3 : nat  - shows type, not value *)

(* Compute works on any expression *)
Compute (fun x => x + 1) 5.  (* = 6 : nat *)

Compute is useful for testing functions — write a function, then Compute a few examples to see if it behaves as expected.

The Power of Induction

Induction is like climbing an infinite ladder:

  1. Base case: Prove for n = 0
  2. Inductive step: If it holds for n, prove it holds for S n
n = 4
n = 3
n = 2
n = 1
n = 0 (base)

Fixpoint vs Definition

In Rocq, Definition creates non-recursive functions, while Fixpoint creates recursive functions.

fixpoint_vs_definition.v
(* Definition: non-recursive, computes directly *)
Definition is_zero (n : nat) : bool :=
  match n with
  | O => true
  | S _ => false
  end.

(* Fixpoint: recursive, calls itself *)
Fixpoint add (n m : nat) : nat :=
  match n with
  | O => m              (* Base case: 0 + m = m *)
  | S p => S (add p m)  (* Recursive: (S p) + m = S (p + m) *)
  end.

Why Fixpoint Needs Termination

Rocq requires all Fixpoint functions to terminate. It checks that recursive calls are made on structurally smaller arguments. In add, the recursive call uses p which is smaller than S p — this guarantees termination!

What does "structurally smaller" mean? A value is structurally smaller if it's a direct subterm. p is structurally smaller than S p because S p was built from p. Similarly, the tail of a list is structurally smaller than the full list.

How Addition Works

Let's trace through 3 + 2:

(*   = S (S ((S O) + S (S O)))      by definition *)
(*   = S (S (S (O + S (S O))))      by definition *)
(*   = S (S (S (S (S O))))          by definition *)
(*   = 5                                          *)

Our First Induction Proofs

induction_proofs.v
Require Import Arith.

(* 0 + n = n is trivial by definition *)
Theorem plus_O_n : forall n : nat, 0 + n = n.
Proof.
  intro n.
  simpl.        (* Simplifies by definition *)
  reflexivity.  (* n = n *)
Qed.

(* n + 0 = n requires induction! *)
Theorem plus_n_O : forall n : nat, n + 0 = n.
Proof.
  intro n.
  induction n as [| n' IHn'].
  - (* Base case: n = 0 *)
    simpl. reflexivity.
  - (* Inductive case: n = S n' *)
    (* IHn' : n' + 0 = n' *)
    simpl.           (* S n' + 0 = S (n' + 0) *)
    rewrite IHn'.    (* S (n' + 0) = S n' *)
    reflexivity.
Qed.

(* Commutativity needs a helper lemma *)
Theorem plus_n_Sm : forall n m : nat, S (n + m) = n + S m.
Proof.
  intros n m.
  induction n as [| n' IHn'].
  - simpl. reflexivity.
  - simpl. rewrite IHn'. reflexivity.
Qed.

Theorem plus_comm : forall n m : nat, n + m = m + n.
Proof.
  intros n m.
  induction n as [| n' IHn'].
  - simpl. rewrite plus_n_O. reflexivity.
  - simpl. rewrite IHn'. rewrite plus_n_Sm. reflexivity.
Qed.

New tactics!

  • simpl — simplify using definitions
  • reflexivity — prove x = x
  • rewrite H — substitute using equation H
  • induction n as [| n' IH] — do induction

Why Does n + 0 = n Need Induction?

This is a common source of confusion! Look at how + is defined — it pattern-matches on the first argument:

  • 0 + n immediately simplifies to n (first case of the match)
  • n + 0 can't simplify unless we know what n is!

For an arbitrary n, we must use induction to consider all possible forms.

Understanding the Induction Pattern

The pattern induction n as [| n' IHn'] looks cryptic, but it's actually quite logical. Let's decode it:

The as Clause

The as [| n' IHn'] part names the variables introduced in each case:

  • [ ... ] — The whole pattern is in square brackets
  • | — Separates the cases (like in match)
  • Before |: names for the base case (n = 0)
  • After |: names for the inductive case (n = S n')

Why Is the Base Case Empty?

The constructor O (zero) takes no arguments, so there's nothing to name. That's why we have [| with nothing before the bar.

The Inductive Case

The constructor S takes one argument (the predecessor), so we name it n'. The induction hypothesis (the property we assume holds for n') is named IHn'.

Before: induction n as [| n' IHn']
n : nat
n + 0 = n

⬇️ induction n as [| n' IHn'].

Base case (n = 0)
(no new hypotheses)
0 + 0 = 0
Inductive case (n = S n')
n' : nat
IHn' : n' + 0 = n'
S n' + 0 = S n'

The Induction Hypothesis is Your Key Tool!

In the inductive case, IHn' : n' + 0 = n' says "the property holds for the predecessor n'". Your job is to use this assumption to prove it holds for S n'. Usually you'll simpl and then rewrite IHn'.

Understanding rewrite

The rewrite tactic uses an equation to substitute in the goal:

  • rewrite H — replaces left side of H with right side (left-to-right)
  • rewrite <- H — replaces right side with left side (right-to-left)
Before rewrite IHn'
IHn' : n' + 0 = n'
S (n' + 0) = S n'

⬇️ rewrite IHn'.

After rewrite IHn'
IHn' : n' + 0 = n'
S n' = S n'

The rewrite replaced n' + 0 with n' (using IHn' left-to-right). Now reflexivity finishes because both sides are identical.

Why simpl + reflexivity Works

simpl unfolds definitions and computes. For S n' + 0, it uses the definition of + to get S (n' + 0). After rewrite IHn', we have S n' = S n', which reflexivity proves because both sides are identical.

Advanced: Strengthening Induction Hypotheses

Sometimes a straightforward induction doesn't work because the induction hypothesis is too weak. The solution is often to prove something stronger.

The Problem: Weak Induction Hypothesis

weak_ih.v
(* Consider proving: rev (rev l) = l *)
(* A naive approach might try: *)

Theorem rev_involutive_attempt : forall (A : Type) (l : list A),
  rev (rev l) = l.
Proof.
  intros A l.
  induction l as [| h t IHt].
  - simpl. reflexivity.
  - simpl.
    (* Goal: rev (rev t ++ [h]) = h :: t *)
    (* IHt: rev (rev t) = t *)
    (* But IHt doesn't help with rev (something ++ [h])! *)
Abort.

(* The fix: prove a helper lemma first *)
Lemma rev_app_distr : forall (A : Type) (l1 l2 : list A),
  rev (l1 ++ l2) = rev l2 ++ rev l1.
(* Now the main theorem follows easily! *)

The generalize dependent Tactic

Sometimes you need to generalize a variable that's already introduced. generalize dependent moves a variable back into the goal:

generalize.v
(* Example: proving a property about two related numbers *)
Theorem le_plus_trans : forall n m p : nat,
  n <= m -> m <= p -> n <= p.
Proof.
  intros n m p Hnm Hmp.
  (* If we do induction on Hnm here, p is fixed in the IH *)
  (* That's too restrictive! *)

  (* Better approach: generalize p before induction *)
  generalize dependent p.
  (* Now the goal is: forall p, m <= p -> n <= p *)
  induction Hnm as [| m' Hnm' IH].
  - (* n = m *) auto.
  - (* n <= m' and m = S m' *)
    intros p Hmp.
    (* Use IH with (pred p) ... this is getting complicated! *)
Abort.

(* Sometimes it's easier to use a library lemma *)
Require Import Lia.
Theorem le_plus_trans' : forall n m p : nat,
  n <= m -> m <= p -> n <= p.
Proof. intros. lia. Qed.

When Induction Gets Stuck

If your induction hypothesis doesn't seem strong enough:

  1. Check which variable you're inducting on. Sometimes a different choice works better.
  2. Look for a helper lemma. The property you need might require proving something more general first.
  3. Try generalizing. Use generalize dependent x to make x part of the induction hypothesis.
  4. Strengthen the statement. Paradoxically, proving forall x y, P x y is sometimes easier than proving P a b for specific a, b.

The assert Tactic for Intermediate Lemmas

You can prove helper lemmas inline using assert:

assert_example.v
Theorem example_with_assert : forall n : nat, n + n = 2 * n.
Proof.
  intro n.
  (* Assert an intermediate fact *)
  assert (H: 2 * n = n + (n + 0)).
  { simpl. reflexivity. }
  (* Now use H *)
  rewrite H.
  rewrite Nat.add_0_r.
  reflexivity.
Qed.

(* assert creates a subgoal to prove the fact,
   then adds it as a hypothesis for the main goal *)
