The List Type

Lists are defined inductively, just like natural numbers:

list.v
(* How list is defined *)
Inductive list (A : Type) : Type :=
  | nil : list A                    (* Empty list *)
  | cons : A -> list A -> list A.   (* Add element to front *)

(* Notation:
   [] for nil
   x :: xs for cons x xs
   [1; 2; 3] for 1 :: 2 :: 3 :: []
*)

Visualizing a list:

1
2
3

[1; 2; 3] = 1 :: 2 :: 3 :: []

Importing the List Library

Before working with lists, we import the standard library:

list_imports.v
Require Import List.      (* Load the List library *)
Import ListNotations.     (* Enable [1; 2; 3] notation *)

(* Without ListNotations, you'd write:
   cons 1 (cons 2 (cons 3 nil))

   With ListNotations, you write:
   [1; 2; 3]                       *)

(* Note: elements are separated by semicolons, not commas! *)
Check [1; 2; 3].   (* list nat *)

Import vs Require Import

Require Import List loads AND imports the module. Import ListNotations is a separate import because ListNotations is a submodule inside List. The notation import is optional but makes code much more readable!

Basic List Functions

list_functions.v
Require Import List.
Import ListNotations.

(* Length of a list *)
Fixpoint length {A : Type} (l : list A) : nat :=
  match l with
  | [] => 0
  | _ :: t => S (length t)
  end.

Compute length [1; 2; 3; 4].  (* = 4 *)

(* Append two lists *)
Fixpoint app {A : Type} (l1 l2 : list A) : list A :=
  match l1 with
  | [] => l2
  | h :: t => h :: app t l2
  end.

(* Notation: l1 ++ l2 *)
Compute [1; 2] ++ [3; 4].  (* = [1; 2; 3; 4] *)

(* Reverse a list *)
Fixpoint rev {A : Type} (l : list A) : list A :=
  match l with
  | [] => []
  | h :: t => rev t ++ [h]
  end.

Compute rev [1; 2; 3].  (* = [3; 2; 1] *)

(* Map a function over a list *)
Fixpoint map {A B : Type} (f : A -> B) (l : list A) : list B :=
  match l with
  | [] => []
  | h :: t => f h :: map f t
  end.

Compute map (fun x => x * 2) [1; 2; 3].  (* = [2; 4; 6] *)

Jeju says:

The {A : Type} with curly braces makes A an implicit argument. Rocq figures it out automatically!

Wildcards in List Patterns

In the length function, notice _ :: t instead of h :: t:

wildcard_list.v
(* We don't use the head, so use _ *)
| _ :: t => S (length t)

(* Equivalent to: *)
| h :: t => S (length t)   (* h is unused - compiler may warn *)

Use _ when you need to match a pattern but don't need the value. It's clearer than naming a variable you never use!

Append Order

l1 ++ l2 puts l1 first, then l2: 

[1; 2] ++ [3; 4] = [1; 2; 3; 4]

The first list's elements come first in the result. Think of ++ as "concatenate l1 followed by l2".

Understanding Implicit Arguments

The curly braces {A : Type} mark an implicit argument. Rocq will infer this argument from context — you don't need to provide it explicitly.

implicit_args.v
(* With implicit argument: {A : Type} *)
Fixpoint length {A : Type} (l : list A) : nat := ...

(* Rocq infers A from the list you pass *)
Compute length [1; 2; 3].       (* A is inferred as nat *)
Compute length [true; false].   (* A is inferred as bool *)

(* Without implicit args, you'd have to write: *)
Compute @length nat [1; 2; 3].  (* @ makes implicits explicit *)

(* Compare to explicit argument: (A : Type) *)
Fixpoint length_explicit (A : Type) (l : list A) : nat := ...

(* You'd always have to provide the type: *)
Compute length_explicit nat [1; 2; 3].

Rule of thumb: Use {...} for type parameters that can be inferred from other arguments. Use (...) for arguments you want callers to explicitly provide.

The @ Symbol: Making Implicits Explicit

Sometimes Rocq can't infer implicit arguments, or you need to specify them explicitly for clarity. The @ symbol turns off implicit argument inference:

at_symbol.v
(* Normal call - Rocq infers type from [1;2;3] *)
Compute length [1; 2; 3].

(* With @ - must provide ALL arguments including implicits *)
Compute @length nat [1; 2; 3].

(* When is @ useful? *)
(* 1. When the list is empty and type can't be inferred: *)
Check @nil nat.           (* [] of type list nat *)

(* 2. In proofs when Rocq gets confused about types *)
(* 3. When you want to be explicit for documentation *)

Lambda Expressions: fun x => ...

The fun keyword creates anonymous functions (lambdas):

lambda.v
(* Basic lambda syntax *)
Check (fun x => x + 1).           (* nat -> nat *)
Check (fun x y => x + y).         (* nat -> nat -> nat *)
Check (fun (x : nat) => x * 2).   (* With type annotation *)

(* Using lambdas with map *)
Compute map (fun x => x * 2) [1; 2; 3].     (* = [2; 4; 6] *)
Compute map (fun x => x + 10) [1; 2; 3].    (* = [11; 12; 13] *)
Compute map (fun b => negb b) [true; false]. (* = [false; true] *)

(* Lambdas can pattern match *)
Compute map (fun p => fst p + snd p) [(1,2); (3,4)]. (* = [3; 7] *)

Lambda vs Named Function

fun x => x + 1 is the same as defining: 

Definition add_one (x : nat) := x + 1.

Lambdas are useful when you need a quick, one-off function, especially as arguments to map, filter, or fold.

Proving Properties of Lists

We use induction on lists, similar to natural numbers:

  • Base case: Prove for []
  • Inductive case: Prove for h :: t, assuming it holds for t
list_proofs.v
Require Import List.
Import ListNotations.

(* Length of append *)
Theorem app_length : forall (A : Type) (l1 l2 : list A),
  length (l1 ++ l2) = length l1 + length l2.
Proof.
  intros A l1 l2.
  induction l1 as [| h t IHt].
  - simpl. reflexivity.
  - simpl. rewrite IHt. reflexivity.
Qed.

(* Append is associative *)
Theorem app_assoc : forall (A : Type) (l1 l2 l3 : list A),
  l1 ++ (l2 ++ l3) = (l1 ++ l2) ++ l3.
Proof.
  intros A l1 l2 l3.
  induction l1 as [| h t IHt].
  - simpl. reflexivity.
  - simpl. rewrite IHt. reflexivity.
Qed.

(* Append with nil *)
Theorem app_nil_r : forall (A : Type) (l : list A),
  l ++ [] = l.
Proof.
  intros A l.
  induction l as [| h t IHt].
  - simpl. reflexivity.
  - simpl. rewrite IHt. reflexivity.
Qed.

Strategy: Induction on l.

  • Base case: map f [] = [], so both sides are 0.
  • Inductive case: simpl shows S (length (map f t)) = S (length t)

The Ss on both sides match, and the induction hypothesis IHt : length (map f t) = length t finishes it.

Exercise 4.3: Map Composition

exercise4_3.v
Require Import List.
Import ListNotations.

Theorem map_map : forall (A B C : Type) (f : A -> B) (g : B -> C) (l : list A),
  map g (map f l) = map (fun x => g (f x)) l.
Proof.
  (* Your proof here! *)

Qed.

Strategy: This is straightforward induction on l.

  • Base case: Both sides simplify to [].
  • Inductive case: After simpl, both sides have the same structure. Apply the IH and you're done!

The key insight: map is structurally recursive, so applying two maps in sequence naturally corresponds to applying the composition.

Advanced: Records and Pairs

Beyond lists, Rocq has records (also called structures) for grouping related data, and built-in pairs for simple two-element tuples.

Pairs: prod

pairs.v
(* The pair type is built-in *)
Check (1, true).           (* nat * bool *)
Check (3, "hello", false). (* nat * string * bool *)

(* Access components with fst and snd *)
Compute fst (1, true).     (* = 1 *)
Compute snd (1, true).     (* = true *)

(* Or use pattern matching *)
Definition swap {A B : Type} (p : A * B) : B * A :=
  match p with
  | (x, y) => (y, x)
  end.

Compute swap (1, true).    (* = (true, 1) *)

(* Destruct in proofs *)
Theorem pair_eq : forall A B (p : A * B), p = (fst p, snd p).
Proof.
  intros A B p.
  destruct p as [a b].   (* Introduces a : A and b : B *)
  simpl.
  reflexivity.
Qed.

Records: Named Fields

records.v
(* Define a record type *)
Record Point := {
  x : nat;
  y : nat
}.

(* Create a record value *)
Definition origin : Point := {| x := 0; y := 0 |}.
Definition p1 : Point := {| x := 3; y := 4 |}.

(* Access fields - automatically generated accessors *)
Compute x p1.      (* = 3 *)
Compute y p1.      (* = 4 *)

(* Records with type parameters *)
Record Pair (A B : Type) := {
  first : A;
  second : B
}.

Arguments first {A B}.
Arguments second {A B}.

Definition my_pair : Pair nat bool := {| first := 42; second := true |}.
Compute first my_pair.   (* = 42 *)

(* Update a field (creates a new record) *)
Definition move_right (p : Point) : Point :=
  {| x := x p + 1; y := y p |}.

Compute move_right p1.   (* {| x := 4; y := 4 |} *)

Records vs Inductive Types

Records are actually syntactic sugar for inductive types with one constructor! The Point record above is equivalent to: 

Inductive Point := mkPoint (x : nat) (y : nat).

Records are preferred when you have a single constructor with named fields. Use regular Inductive types when you have multiple constructors (like list with nil and cons).

Higher-Order List Functions

Let's look at some powerful functions that take other functions as arguments:

higher_order.v
Require Import List.
Import ListNotations.

(* FILTER: keep elements satisfying a predicate *)
Compute filter (fun n => n  n  5  n * n)
      (filter (fun n => Nat.even n) l)).

Compute sum_even_squares [1;2;3;4;5;6].   (* 4 + 16 + 36 = 56 *)
